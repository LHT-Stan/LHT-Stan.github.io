显示Intent（Explicit Intent）是指明确指定目标组件的Intent。它通常用于启动应用程序内的特定组件，例如启动另一个Activity、启动Service或发送广播。显示Intent通过指定目标组件的类名或组件的包名和类名来明确指定要启动的组件。

隐式Intent（Implicit Intent）是没有明确指定目标组件的Intent。它只描述了要执行的操作、要传递的数据以及可能的类别等信息。隐式Intent允许系统根据给定的操作和数据来找到最佳的组件来处理该Intent。系统会根据Intent中的信息，查找能够响应该Intent的组件，并将该Intent发送到最合适的组件。

隐式Intent的匹配是通过Intent-filter来实现的。它描述了组件能够响应的意图。Intent-filter包含以下三个主要部分：

  1. Action（动作）：指定要执行的操作，例如”android.intent.action.VIEW”表示查看操作、”android.intent.action.SEND”表示发送操作等。一个Intent-filter可以指定一个或多个Action。
  2. Category（类别）：指定与组件相关的额外信息。例如，一个Activity可以指定”android.intent.category.DEFAULT”类别来指示它是默认的操作，或者指定”android.intent.category.BROWSABLE”类别来指示它可以被浏览器打开。
  3. Data（数据）：指定Intent操作要处理的数据。数据可以是一个URI（统一资源标识符），例如网址、文件路径等。通过匹配Intent-filter中指定的数据，系统可以找到能够处理该数据的组件。

2、Service（服务）

简介：

服务（Service）是 Android 平台提供的一种组件，用于在后台执行长时间运行的操作或处理一些独立于界面的任务。服务在 Android 应用程序中具有以下作用：

  1. 后台任务执行：服务可以在后台执行长时间运行的任务，而不会受到用户界面的限制。例如，下载文件、播放音乐、处理数据同步等操作可以放在服务中进行，而不会中断或影响用户的交互体验。
  2. 资源共享：服务可以作为一个独立的组件，供其他组件共享资源或数据。多个组件可以绑定到同一个服务上，并通过服务提供的接口进行通信和数据交换。这样可以避免数据冗余和资源重复创建。
  3. 跨组件通信：服务可以作为组件之间的中介，实现跨组件的通信。不同组件（如 Activity、Fragment、BroadcastReceiver）可以通过绑定服务或发送广播与服务进行通信，实现数据传递和功能调用。
  4. 前台通知：服务可以将自身提升为前台服务，通过发送通知显示在状态栏中，并且在通知被移除前一直处于运行状态。这在需要保持服务持续运行且用户可见的场景中非常有用，如音乐播放器、定位服务等。
  5. 生命周期管理：服务具有自己的生命周期，可以通过 onStartCommand() 方法或 onBind() 方法来启动和停止服务。这样可以根据应用程序的需求来控制服务的生命周期，并确保服务在适当的时候被启动或停止。

总结：

  • 服务允许在后台执行长时间运行的任务，不受用户界面限制。
  • 服务可以作为资源共享和跨组件通信的中介。
  • 服务可以通过前台通知将自身提升为可见状态。
  • 服务具有自己的生命周期，可以根据需求进行启动和停止。

    • 启动式服务适用于执行一次性任务、执行后台操作、播放音乐等不需要与调用者进行持续交互的场景。
    • 启动式服务需要在任务完成后调用 stopSelf() 或 stopService() 方法来停止服务。
  2. 绑定式服务（Bound Service）：
    • 绑定式服务通过调用 bindService() 方法与调用者绑定，并与调用者共享生命周期。
    • 绑定式服务与调用者建立关联后，可以通过返回的 Binder 对象与调用者进行交互，调用者可以调用服务中的方法。
    • 绑定式服务适用于需要与调用者进行交互、共享数据或提供特定功能接口的场景。
    • 绑定式服务需要在调用者不再需要服务时调用 unbindService() 方法解除绑定。

总结：

  • 启动式服务适用于一次性任务、后台操作、长时间运行的任务，不需要与调用者持续交互。
  • 绑定式服务适用于需要与调用者交互、共享数据或提供特定功能接口的场景。

3、BroadcastReceiver（广播）

广播（Broadcast）是 Android 中一种用于在应用程序内或应用程序之间传递消息和事件的机制。通过广播，应用程序可以发送消息（广播）并让其他组件（接收者）接收并处理这些消息。以下是广播的一些要点：

  1. 广播发送者（Broadcast Sender）：
    • 广播发送者是发送广播的组件，可以是应用程序中的任何组件，如 Activity、Service、BroadcastReceiver 等。
    • 广播发送者通过调用 sendBroadcast()、sendOrderedBroadcast() 或 sendStickyBroadcast() 等方法发送广播。
    • 广播发送者需要创建一个 Intent 对象，并将其用作广播的载体，可以在 Intent 中携带附加数据。
  2. 广播接收者（Broadcast Receiver）：
    • 广播接收者是接收广播的组件，可以是应用程序内的组件或其他应用程序的组件。
    • 广播接收者需要继承自 BroadcastReceiver 并实现 onReceive() 方法，用于接收和处理广播。
    • 广播接收者可以通过在 AndroidManifest.xml 文件中声明，或动态注册（通过代码注册）来注册。
  3. 广播类型：
    • 标准广播（Normal Broadcast）：以异步方式传递给所有匹配的接收者，接收者之间无序接收广播。
    • 有序广播（Ordered Broadcast）：以同步方式传递给所有匹配的接收者，并按照优先级依次接收广播。
    • 粘性广播（Sticky Broadcast）：被发送后仍然保留在系统中，允许接收者在稍后注册时接收广播。
  4. 广播接收者的注册：
    • 静态注册：在 AndroidManifest.xml 文件中声明广播接收者，并指定其接收的广播类型和优先级。
    • 动态注册：通过在代码中调用 registerReceiver() 方法来动态注册广播接收者，并在不需要接收广播时调用 unregisterReceiver() 方法解除注册。

广播机制允许应用程序内部或不同应用程序之间的组件进行通信，可以用于实现一些事件的广播通知、系统级别的广播监听、应用程序之间的消息传递等。广播是 Android 中一种重要的组件通信方式，提供了灵活的消息传递机制。

4、Content Provider（数据分享）

简介：

Content Provider（内容提供者）是 Android 中的一种组件，用于管理应用程序之间共享的数据，并提供对这些数据的安全访问。Content Provider 允许应用程序之间通过标准化的接口进行数据交换和共享，同时提供了数据的增删改查等操作。

以下是 Content Provider 的一些要点：

  1. 数据共享：Content Provider 提供了一种机制，使得应用程序可以共享自己的数据给其他应用程序访问。这些数据可以是应用程序内部的数据，也可以是来自外部存储、数据库或网络的数据。
  2. 数据访问控制：Content Provider 可以通过权限设置和 URI 权限匹配等方式对数据进行访问控制，确保只有具备相应权限的应用程序才能访问和修改数据。
  3. 标准化接口：Content Provider 使用标准化的接口和操作方法（如查询、插入、更新和删除）来对数据进行操作，这样不同的应用程序可以以统一的方式访问和操作数据。
  4. URI（Uniform Resource Identifier）：URI 是 Content Provider 中用于标识和定位数据的唯一标识符。应用程序可以通过指定 URI 来访问 Content Provider 中的数据。
  5. 内容观察者（Content Observer）：Content Provider 支持内容观察者模式，即应用程序可以注册内容观察者来监听数据的变化，当数据发生变化时，内容观察者将收到通知。
  6. 隐式访问和显式访问：Content Provider 可以通过隐式访问和显式访问两种方式来提供数据访问。隐式访问使用 URI 匹配规则，可以由不同的应用程序访问同一个 Content Provider。显式访问使用指定的 Content Provider 名称，只能由特定的应用程序访问。

Content Provider 提供了一种方便和安全的方式来共享和访问数据，它在 Android 应用程序之间起到了数据交换和共享的桥梁作用。通过 Content Provider，应用程序可以将自身的数据开放给其他应用程序使用，实现了数据的互通性和复用性。

八、线程通信机制

Handler 和 Looper 是 Android 中用于实现线程间通信的关键组件。它们与消息队列（Message Queue）密切相关。

  1. 消息队列（Message Queue）：消息队列是一个先进先出（FIFO）的数据结构，用于存储需要在特定线程中处理的消息（Message）。每个线程都有一个关联的消息队列，用于存储待处理的消息。
  2. Looper：Looper 是一个线程本地的消息循环器，通过循环从消息队列中取出消息，并将其分发给相应的 Handler 进行处理。每个线程只能有一个 Looper，并且只能在 Looper 所在的线程中调用 Looper 的相关方法。
  3. Handler：Handler 是与特定 Looper 关联的对象，用于发送和处理消息。它可以将消息发送到消息队列中，也可以从消息队列中取出消息并进行处理。Handler 可以关联到特定的线程，并在该线程的消息循环中进行操作。

消息发送和接收处理的方法包括：

  • 发送消息：可以使用 Handler 的 sendMessage() 或 post() 方法发送消息到消息队列中。例如，handler.sendMessage(message) 或 handler.post(runnable)。
  • 处理消息：Handler 可以重写 handleMessage() 方法来处理接收到的消息。当消息到达消息队列并被取出时，将调用 Handler 的 handleMessage() 方法来处理该消息。

*64. (编程题,15.0分)请根据要求写出代码：界面上放置两个button和一个文本输入框，其中一个按钮的功能是将文本输入框的内容保存到本地SharedPrefence中，另外一个按钮是清除SharedPrefence中的数据。*

//示例仅供参考

package com.example.test

import android.content.Context
import androidx.appcompat.app.AppCompatActivity
import android.os.Bundle
import android.widget.Button
import android.widget.EditText

class ExamActivity : AppCompatActivity() {
private lateinit var btSave:Button
private lateinit var btClear:Button
private lateinit var tvContent:EditText
override fun onCreate(savedInstanceState: Bundle?) {
super.onCreate(savedInstanceState)
setContentView(R.layout.activity_exam_activiy)
btSave=findViewById(R.id.bt_save)
btClear=findViewById(R.id.bt_clear)
tvContent=findViewById(R.id.tv_content)
val sharedPrefence=getSharedPreferences(“ContentData”,Context.MODE_PRIVATE)
btSave.setOnClickListener {
val content=tvContent.text.trim().toString()
val editor=sharedPrefence.edit()
editor.putString(“content”,content)
editor.commit()
}
btClear.setOnClickListener {
val editor=sharedPrefence.edit()
editor.clear()
editor.commit()
}
}
}
